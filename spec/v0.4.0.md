# zjsh v0.4.0 Spec

## Status

The v0.3.2 spec now covers the `.` ordering change and the `project.startup` deprecation path.

This document only covers the remaining v0.4.0 session-policy work.

## Overview

v0.4.0 focuses on two session-policy changes:

- Add a config policy for disposable Zellij sessions.
- Add an explicit project policy for recreating a project session on connect.

Behavior not mentioned in this spec remains as defined by v0.3.2.

## Main Changes

- Config `defaults` are clarified as global defaults for zjsh-managed behavior, not project-only defaults.
- New config field: `resurrectable`.
- `defaults.resurrectable` controls whether newly-created zjsh-managed sessions are saved for later resurrection.
- `project.resurrectable` overrides `defaults.resurrectable` for that project only.
- `resurrectable false` maps to Zellij session serialization being disabled for the created session.
- New project-only config field: `recreate_on_connect`.
- `project.recreate_on_connect true` replaces the previous resurrection-only recreate behavior with a broader project recreate policy that discards an existing live or resurrectable project session before recreating it.
- `project.startup` is removed from the v0.4.0 config surface; use a Zellij `layout` or `layout_file` for startup commands.
- New session creation is unified through background creation before connecting.

## Terms

### zjsh-managed session

A zjsh-managed session is a Zellij session that zjsh creates or recreates from a configured project, a zoxide path entry, or the built-in current-directory entry (`.`).

### Pure Zellij session entry

A pure Zellij session entry is an entry discovered from existing Zellij state that is not backed by a configured project, zoxide path entry, or the built-in `.` entry.

Examples:

- an existing live Zellij session that does not match any configured project, zoxide path, or current-directory entry;
- an existing resurrectable Zellij session that does not match any configured project, zoxide path, or current-directory entry.

Pure Zellij session entries are not considered zjsh-managed creation targets.

### Matching project session

A matching project session is any Zellij session whose name equals the configured project's effective session name.

Session name priority:

1. `project.session`
2. `project.name`

This match is name-based. If a manually-created Zellij session uses the same name as a configured project with `recreate_on_connect true`, selecting that project may force-delete the manually-created session before recreating the project session.

## Global Defaults

Config `defaults` define fallback behavior for zjsh-managed entries.

Mechanism:

- A field under `defaults` applies to every zjsh-managed entry where that field is meaningful.
- A project-level field overrides the corresponding global default for that project only.
- If a global default is absent, zjsh uses its built-in default for that field.

For v0.4.0, the newly-added global default is `defaults.resurrectable`.

## Config `resurrectable`

`resurrectable` is a user-facing policy for whether newly-created zjsh-managed sessions should be saved for later resurrection.

This option does not delete or modify existing serialized sessions. It only controls whether newly-created zjsh-managed sessions should be serialized by Zellij. Existing live or resurrectable state is discarded only by `recreate_on_connect`.

Example:

```kdl
defaults {
  resurrectable false
}

project "api" {
  path "~/work/api"
  session "api"
  resurrectable true
}

project "scratch" {
  cwd true
  session "scratch"
  resurrectable false
  recreate_on_connect true
}
```

Effective policy:

* configured project: `project.resurrectable` → `defaults.resurrectable` → built-in `true`
* zoxide path entry: `defaults.resurrectable` → built-in `true`
* built-in `.` entry: `defaults.resurrectable` → built-in `true`
* pure Zellij session entry: not applicable

Semantics:

* `resurrectable true` preserves Zellij's normal session resurrection behavior.
* `resurrectable false` disables saving newly-created sessions for later resurrection.
* Pure Zellij session entries do not use `defaults.resurrectable` because connecting to them does not create a new zjsh-managed session.

Purpose: this exposes Zellij's session serialization behavior with a product-level name, without exposing `session_serialization` as the zjsh config field.

## Config `recreate_on_connect`

`recreate_on_connect` is a project-only policy for always recreating a configured project session when connecting to that project.

Example:

```kdl
project "scratch" {
  cwd true
  session "scratch"
  resurrectable false
  recreate_on_connect true
}
```

Effective policy:

* configured project: `project.recreate_on_connect` → built-in `false`
* zoxide path entry: not applicable
* built-in `.` entry: not applicable
* pure Zellij session entry: not applicable

Semantics:

* `recreate_on_connect true` force-deletes an existing live or resurrectable session for the project, then recreates it from project config.
* If no matching live or resurrectable session exists, connect creates the project session normally.
* The replacement session uses the effective project creation policy, including `resurrectable`.
* `recreate_on_connect false` preserves the existing connect behavior: live sessions attach/switch, and resurrectable sessions resurrect.

Safety rule:

* `recreate_on_connect` is intentionally not available under `defaults` because a global setting that force-deletes live sessions is too dangerous.
* `defaults.recreate_on_connect` is invalid config and must fail instead of being ignored.

Purpose: this gives explicitly configured disposable projects, such as scratch workspaces, a reliable fresh-session behavior without applying destructive behavior globally.

Destructive UX rule:

* `recreate_on_connect true` is intentionally explicit and project-local because it can delete live sessions.
* v0.4.0 does not add a separate destructive indicator to list output.
* JSON output should expose enough entry policy data for external selectors to warn users if desired.

### Replacement for `restart_on_resurrection`

v0.4.0 replaces `restart_on_resurrection` with `recreate_on_connect`.

Differences:

* `restart_on_resurrection` only handled existing resurrectable state.
* `recreate_on_connect` handles both existing live project sessions and existing resurrectable project sessions.

Compatibility rule:

* `restart_on_resurrection` is not accepted in v0.4.0 config.
* Config containing `restart_on_resurrection` must fail with a clear migration error instead of being silently ignored.

## Intended Use

`resurrectable false` is intended for disposable or scratch workspaces.

With this config:

```kdl
defaults {
  resurrectable false
}
```

new sessions launched from zoxide paths and `.` are disposable by default.

Configured long-lived projects can opt back in to resurrection:

```kdl
project "api" {
  path "~/work/api"
  resurrectable true
}
```

Expected behavior for a disposable target:

* connecting creates a normal Zellij session;
* when that session is killed or exits and no live session remains, it should not leave serialized resurrectable state behind;
* a later connect creates a fresh workspace when no live session exists.

## Launch Behavior

`resurrectable` only affects newly-created zjsh-managed sessions.

Existing sessions:

* Existing live sessions attach or switch as before when no recreate policy applies.
* Existing resurrectable sessions resurrect as before when no recreate policy applies.
* Existing live or resurrectable sessions for a project with `recreate_on_connect true` are force-deleted before replacement creation.
* Replacement sessions use the effective project creation policy.

Recreate operation order:

1. Check whether recreating would delete the current Zellij session that is running zjsh.
2. Force-delete the existing matching live or resurrectable project session.
3. Verify that no matching live or resurrectable session state remains.
4. Create the replacement session in the background.
5. Connect to the replacement session.

Failure semantics:

* If zjsh is running inside the same live session that `recreate_on_connect` would force-delete, zjsh must fail explicitly before invoking `zellij delete-session --force`.
* If force-deleting the existing session fails, zjsh must abort and must not create a replacement session.
* If matching live or resurrectable state remains after force-delete, zjsh must abort and must not create a replacement session.
* If background creation fails, zjsh must abort and must not attempt to attach or switch.
* Force-deleting a live session may disconnect other clients attached to that session; this is expected behavior for an explicitly configured `recreate_on_connect true` project.

### Unified Creation Flow

New session creation must not rely on `zellij action switch-session` implicitly creating missing sessions.

Instead, every newly-created zjsh-managed session uses a two-step flow:

1. Create the session in the background.
2. Connect to the now-existing session.

Creation step, shown schematically:

```sh
zellij attach --create-background <session> options ...
```

Connection step:

* outside Zellij: `zellij attach <session>`
* inside Zellij: `zellij action switch-session <session>`

Purpose:

* keep new-session creation behavior consistent inside and outside Zellij;
* keep `switch-session` responsible only for switching to an already-created session;
* provide a single place to apply creation-time options such as `--session-serialization false`.

Creation paths that must support `resurrectable false`:

* configured project with `path`;
* configured project with `cwd true`;
* zoxide path entry;
* built-in `.` entry;
* named layout creation;
* layout file creation.

### Creation Inputs

Background creation should use the same effective creation input that the old direct creation path would have used.

| Creation type                      | Background creation input                  |
| ---------------------------------- | ------------------------------------------ |
| configured project with `path`     | project path as cwd                        |
| configured project with `cwd true` | runtime cwd                                |
| zoxide path entry                  | selected zoxide path as cwd                |
| built-in `.` entry                 | runtime cwd                                |
| named layout project               | selected layout name plus effective cwd    |
| layout file project                | selected layout file plus effective cwd    |

CLI mapping:

* Recreate deletion uses `zellij delete-session --force <session>` for both live and resurrectable project sessions.
* For background creation paths that use `zellij attach --create-background ... options ...`, append `--session-serialization false` when the effective policy is `resurrectable false`.
* Named layout and layout file creation should pass the selected layout during the background creation step.
* Plain path, zoxide path, and `.` creation should pass the selected cwd during the background creation step.
* `resurrectable true` should not emit an explicit `--session-serialization true` unless required by implementation constraints.

Implementation note:

* Implementation must verify that `zellij delete-session --force <session>` removes live sessions, resurrectable sessions, and any matching residual state before replacement creation.
* Implementation must verify that `zellij attach --create-background` supports the required combination of cwd, named layout, layout file, and session serialization options.
* If a required Zellij CLI combination is unsupported, zjsh should fail explicitly instead of silently creating a session with behavior that contradicts the effective `resurrectable` policy.

When `recreate_on_connect true` applies, zjsh first force-deletes the existing live or resurrectable project session, then uses the same background creation flow for the replacement session.

## Config Parser

New known `defaults` field:

* `resurrectable`

New known `project` field:

* `resurrectable`
* `recreate_on_connect`

Removed `project` field:

* `startup`

Compatibility rules:

* If `defaults.resurrectable` is absent, use built-in `true`.
* If `project.resurrectable` is absent, use the effective global default policy.
* If `project.recreate_on_connect` is absent, use built-in `false`.
* `project.startup` is not supported in v0.4.0. Users should express startup commands in Zellij layouts and reference them with `layout` or `layout_file`.

Failure rules:

* non-boolean `defaults.resurrectable` fails;
* non-boolean `project.resurrectable` fails;
* non-boolean `project.recreate_on_connect` fails;
* `resurrectable` nodes with missing or multiple values fail;
* `recreate_on_connect` nodes with missing or multiple values fail;
* `defaults.recreate_on_connect` fails because destructive recreate behavior is not allowed as a global default.
* `project.startup` fails because startup command generation is removed from v0.4.0 config.
* `restart_on_resurrection` fails with a migration error because v0.4.0 replaces it with `project.recreate_on_connect`.

Existing v0.3.0 parser compatibility and failure rules remain unchanged except for the explicit `restart_on_resurrection` migration failure.

## Entry Data

Entries that can create or recreate zjsh-managed sessions carry the effective creation/recreation policy into launch-time entry data.

Mechanism:

* Configured project entries carry the effective policy from project override, global default, or built-in default.
* Zoxide path entries carry the effective policy from global default or built-in default.
* The built-in `.` entry carries the effective policy from global default or built-in default.
* When a configured project merges with a live or resurrectable Zellij session, preserve the project policy in the merged entry.
* When a configured project merges with the runtime cwd or a zoxide path, preserve the project policy in the merged entry.
* Pure Zellij session entries do not receive `defaults.resurrectable`.
* Only configured project entries can carry `recreate_on_connect true`.
* Zoxide path entries and the built-in `.` entry carry no applicable destructive recreate policy.
* Pure Zellij session entries carry no zjsh-managed creation or recreation policy.

Implementation guidance:

* Entry data should distinguish between `resurrectable false`, project-level `recreate_on_connect false`, and "not applicable".
* A nullable creation policy is preferred over a bare boolean field.
* Pure Zellij session entries should not appear as `resurrectable false`; they should have no zjsh-managed creation policy.

Example model:

```go
type CreationPolicy struct {
    Resurrectable bool
    // nil means destructive recreate is not applicable for this entry.
    RecreateOnConnect *bool
}

type Entry struct {
    // nil means this entry has no zjsh-managed creation/recreation policy.
    CreationPolicy *CreationPolicy
}
```

## Test Requirements

### Config

* missing `resurrectable` defaults to `true`;
* `defaults.resurrectable true` parses successfully;
* `defaults.resurrectable false` parses successfully;
* `project.resurrectable true` parses successfully;
* `project.resurrectable false` parses successfully;
* `project.resurrectable` overrides `defaults.resurrectable`;
* `project.recreate_on_connect true` parses successfully;
* `project.recreate_on_connect false` parses successfully;
* missing `project.recreate_on_connect` defaults to `false`;
* non-boolean `defaults.resurrectable` fails;
* non-boolean `project.resurrectable` fails;
* non-boolean `project.recreate_on_connect` fails;
* missing-value or multiple-value `resurrectable` nodes fail;
* missing-value or multiple-value `recreate_on_connect` nodes fail;
* `defaults.recreate_on_connect` fails;
* `project.startup` fails with a clear removal error;
* `project.restart_on_resurrection` fails with a migration error;
* `defaults.restart_on_resurrection` fails with a migration error;
* sample configs do not rely on `project.startup`;
* existing sample configs updated to v0.4.0 fields keep parsing successfully.

### Entry Data

* configured project entries carry the effective `resurrectable` policy;
* zoxide path entries carry the effective `resurrectable` policy;
* the built-in `.` entry carries the effective `resurrectable` policy;
* project override is preserved after aggregation;
* project policy is preserved when merging with a live session;
* project policy is preserved when merging with a resurrectable session;
* project policy is preserved when merging with a zoxide path;
* project policy is preserved when merging with the runtime cwd;
* project `recreate_on_connect` policy is preserved after aggregation and merge;
* explicit project `recreate_on_connect false` is distinguishable from not-applicable recreate policy;
* zoxide path entries do not receive `recreate_on_connect` policy;
* the built-in `.` entry does not receive `recreate_on_connect` policy;
* pure Zellij session entries do not receive the global `resurrectable` policy;
* pure Zellij session entries do not receive any zjsh-managed creation or recreation policy.

### Launch

* creating a project with `resurrectable false` emits Zellij session serialization disabled behavior;
* creating a project with `resurrectable true` preserves existing default resurrection behavior;
* new session creation uses background creation before connecting;
* inside-Zellij new session creation does not rely on `switch-session` implicitly creating missing sessions;
* after background creation, outside-Zellij connect attaches to the created session;
* after background creation, inside-Zellij connect switches to the created session;
* `defaults.resurrectable false` applies to projects without a project override;
* `defaults.resurrectable false` applies to zoxide path creation;
* `defaults.resurrectable false` applies to built-in `.` creation;
* `recreate_on_connect true` plus an existing live project session force-deletes the live session, creates a replacement session, and connects to it;
* `recreate_on_connect true` plus an existing resurrectable project session force-deletes old resurrectable state, creates a replacement session, and connects to it;
* `recreate_on_connect true` plus both matching live and resurrectable state leaves no matching old state before background creation;
* `recreate_on_connect true` plus `resurrectable false` creates a fresh non-resurrectable replacement session;
* `recreate_on_connect false` plus an existing live project session attaches or switches without deleting the session;
* `recreate_on_connect false` plus an existing resurrectable project session resurrects without deleting old state;
* `recreate_on_connect true` with no existing session creates normally without attempting removal;
* `recreate_on_connect true` operations occur in order: current-session safety check, force-delete existing session, verify no old state remains, background create, connect;
* recreate deletion uses `zellij delete-session --force <session>` for both live and resurrectable sessions;
* failed existing-session deletion aborts without background creation;
* remaining old state after force-delete aborts without background creation;
* failed background creation aborts without attach or switch;
* attempting to recreate the current live session from inside that session fails explicitly before force-delete;
* named layout creation supports `resurrectable false`;
* layout file creation supports `resurrectable false`;
* attaching or switching to an existing live session is not changed by `resurrectable`;
* attaching or switching to an existing live session is not changed even when the merged project policy is `resurrectable false`;
* resurrecting an already-saved session without recreate policy is not changed by `resurrectable`.

### CLI connect

* connecting to a configured scratch project with `resurrectable false` emits Zellij session serialization disabled behavior;
* connecting to a zoxide path with `defaults.resurrectable false` emits Zellij session serialization disabled behavior;
* connecting to `.` with `defaults.resurrectable false` emits Zellij session serialization disabled behavior;
* connecting to a configured project when no `resurrectable` value is configured preserves the existing default resurrection behavior;
* JSON output exposes entry policy data needed for external selectors to identify `recreate_on_connect true` entries.

## Non-Goals

* Do not expose `session_serialization` as the zjsh config field name.
* Do not expose `recreate_on_connect` under `defaults`.
* Do not expose project startup commands through `project.startup`; use Zellij layouts instead.
* Do not keep `restart_on_resurrection` as the v0.4.0 recreate policy.
* Do not replace recreate behavior with `zellij attach --forget` in v0.4.0 because v0.4.0 uses explicit discard plus unified background creation for consistent behavior.
* Do not delete existing live or serialized sessions merely because `resurrectable false` is configured.
* Do not apply `defaults.resurrectable` to pure Zellij session entries.
* Do not apply `recreate_on_connect` to zoxide path entries, the built-in `.` entry, or pure Zellij session entries.
