# zjsh v0.3.0 Spec

## Overview

v0.3.0 makes `zjsh` a more complete unified entrypoint for Zellij workflows.

Main changes:

- `zjsh list` includes `.` as a built-in current-directory target.
- Config projects can use `cwd true` to define reusable current-directory templates.
- Active sessions sort above static projects while project config remains authoritative after merge.
- Config parsing uses the selected KDL parser, `github.com/sblinch/kdl-go`.
- `zoxide` is optional; missing or failing zoxide data no longer blocks `list` or `connect`.
- `doctor` reports required failures and optional-source warnings separately.

## Entry Sources

`zjsh list` aggregates entries from these sources:

- configured projects;
- live Zellij sessions;
- resurrectable Zellij sessions;
- zoxide paths, when zoxide is available;
- built-in current-directory entry (`.`).

`zellij` is required. Config and zoxide are optional sources.

## Current Directory Entry: `.`

`zjsh list` always includes a built-in `.` entry that represents the runtime current working directory.

Mechanism:

- `.` represents the absolute runtime current working directory.
- Plain list output renders this entry as `.`.
- Icon list output renders this entry as an icon label whose selector value is `.`.
- `zjsh connect .` resolves to this entry.
- Session naming for `.` follows the existing path-based rule:
  - use the path basename when available;
  - append the existing short path hash if the desired name is reserved;
  - preserve existing fallback behavior for invalid, empty, or special basenames.

Purpose: this provides a config-free fallback target for launching from the current directory.

### `.` and zoxide Overlap

If the runtime current directory also appears in zoxide results:

- merge cwd and matching zoxide path into one visible entry;
- render the merged entry as `.`;
- do not also render the matching absolute zoxide path;
- ordinary zoxide paths that do not equal cwd remain visible.

## Config `cwd true` Project Templates

Project config supports `cwd true` for reusable project templates whose effective path is the runtime current directory.

Example:

```kdl
project "here" {
  cwd true
  session "my-session"
  startup "nvim ."
  layout "compact"
}
```

Mechanism:

- `cwd true` projects are normal project entries.
- They display by project name, not as `.`.
- They are always included in `zjsh list`.
- Their effective path is the runtime cwd.
- They support `startup`, `layout`, and `layout_file`.
- They participate in merge and sorting using the same rules as static projects.
- `doctor` does not perform static path existence checks for `cwd true` project paths.

Purpose: `.` is intentionally minimal; `cwd true` provides a named template that can use project features such as session overrides, startup commands, layouts, layout files, and resurrection policy.

### Path Validation

A project must define exactly one path mode:

- static `path`; or
- `cwd true`.

Invalid cases:

- both `path` and `cwd true` are defined;
- neither `path` nor `cwd true` is defined.

### Session Naming

For `cwd true` projects, session name priority is:

1. `project.session`
2. `project.name`

The runtime cwd basename is not used as the default session name for `cwd true` projects.

## Aggregation and Merge

Aggregation-time merge remains enabled.

Mechanism:

- Project + matching live session merge into one entry.
- Project + matching resurrectable session merge into one entry.
- The resulting entry preserves project data and session state.
- Ordinary zoxide path entries remain independent actions.
- The only zoxide merge exception is cwd overlap with `.`.

When a project and session merge, project data remains primary.

Project data to preserve:

- project name;
- effective path;
- session override;
- shell/default shell behavior;
- startup command;
- layout;
- layout file;
- resurrection policy.

Session data to preserve:

- session name;
- session state (`live` or `resurrectable`).

## Display Ordering

Merge priority and display ordering are separate concepts:

- merge priority decides which source owns the resulting entry data;
- display ordering decides where the resulting entry appears in `zjsh list`.

Global display order:

1. live session
2. project
3. resurrectable session
4. zoxide path
5. `.` current-directory entry

Merged entry ordering:

- `project + live session` displays in the live session section.
- `project + resurrectable session` displays in the project section.
- pure resurrectable session displays in the resurrectable session section.

Within each display group, preserve stable source order where possible:

- config order for projects;
- Zellij output order for sessions;
- zoxide output order for zoxide paths.

## zoxide Behavior

`zoxide` is an optional source, not a required dependency.

Mechanism:

- Missing `zoxide` produces no zoxide entries and no list/connect failure.
- Failed `zoxide query -l` produces no zoxide entries and no list/connect failure.
- zoxide entries are included only when data is available.
- zoxide paths display after sessions/projects/resurrectable sessions and before `.`.
- Ordinary zoxide path entries stay independent unless they overlap with cwd.

## Doctor Behavior

`zjsh doctor` reports required failures and optional-source warnings separately.

Mechanism:

- Missing `zellij` is `FAIL`.
- Missing `zoxide` is `WARN`.
- Missing config file is not a failure.
- Existing config with invalid KDL or invalid required values is `FAIL`.
- Static project paths are checked for existence.
- `cwd true` project paths are not statically checked.
- `layout_file`, when configured, is checked for path expansion and existence.

Dependency semantics:

- `zellij`: required runtime dependency.
- `zoxide`: optional list source.
- config file: optional project source.

## Config Parser

Config parsing uses the selected KDL parser:

- `github.com/sblinch/kdl-go`

The KDL parser handles syntax; zjsh handles business validation.

### Supported Config Nodes

Known top-level nodes:

- `defaults`
- `project`

Known `defaults` fields:

- `shell`
- `restart_on_resurrection`
- `icon_project`
- `icon_session`
- `icon_resurrectable`
- `icon_path`

Known `project` fields:

- `path`
- `cwd`
- `session`
- `startup`
- `layout`
- `layout_file`
- `restart_on_resurrection`

### Compatibility Rules

- Known fields are parsed normally.
- Unknown fields are ignored.
- Unknown top-level nodes are ignored.
- Missing config file returns default config.

### Failure Rules

Config parsing fails on correctness-impacting issues:

- invalid KDL syntax;
- invalid value type, such as non-boolean `cwd` or `restart_on_resurrection`;
- project missing both `path` and `cwd true`;
- project defining both `path` and `cwd true`;
- path expansion failure for static `path` or `layout_file`.

## Test Requirements

### Aggregation

- live session sorts before project;
- project sorts before pure resurrectable session;
- pure resurrectable session sorts before zoxide;
- zoxide sorts before `.`;
- `project + live session` merges into one entry and sorts in the live session section;
- `project + resurrectable session` merges into one entry and stays in the project section;
- zoxide path remains an independent action;
- cwd and matching zoxide path merge into one `.` entry;
- `cwd true` project appears in the project section when not live;
- `cwd true` project with live session sorts in the live session section;
- `cwd true` project with resurrectable session stays in the project section.

### CLI list/connect

- `zjsh list` includes `.`;
- `zjsh list -i` includes an icon label for `.`;
- `zjsh connect .` connects using runtime cwd;
- selecting an icon label whose selector value is `.` works;
- ordinary zoxide path labels still resolve;
- `cwd true` project displays by project name;
- connecting to a `cwd true` project uses runtime cwd as the effective path;
- `cwd true` project uses `project.session` before `project.name`;
- `cwd true` project does not use runtime cwd basename as default session name.

### Config

- existing sample configs parse successfully with `kdl-go`;
- unknown fields are ignored;
- unknown top-level nodes are ignored;
- invalid KDL syntax fails;
- invalid boolean values fail;
- project with static `path` remains valid;
- project with `cwd true` and no `path` is valid;
- project with both `path` and `cwd true` fails;
- project with neither `path` nor `cwd true` fails;
- `layout` and `layout_file` are allowed with `cwd true`;
- home path expansion behavior is preserved for static paths and layout files.

### zoxide Optional Behavior

- missing zoxide returns no paths and no error;
- failed `zoxide query -l` returns no paths and no error;
- list/connect still work without zoxide;
- doctor reports missing zoxide as `WARN`, not `FAIL`.

### Doctor

- missing zellij fails doctor;
- missing zoxide warns but does not fail doctor;
- missing config reports optional missing without failure;
- invalid existing config fails doctor;
- `cwd true` project does not perform static project path existence check;
- static `path` project keeps path existence checks.

## Non-Goals

- Do not introduce a built-in selector UI.
- Do not make zoxide required.
- Do not make unknown config fields fatal.
- Do not merge ordinary zoxide paths into projects beyond the cwd overlap rule.
- Do not make `cwd true` project session names depend on runtime cwd basename by default.

## Open Questions

No open product behavior remains for v0.3.0 scope.
