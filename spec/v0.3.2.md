# zjsh v0.3.2 Spec

## Scope

v0.3.2 is a small delta from v0.3.0.

Behavior not changed by this document remains as defined by v0.3.0.

## Summary of Changes

- Move the built-in current-directory entry (`.`) above ordinary zoxide path entries.
- Make `.` always remain an independent entry; it never participates in aggregation-time merge.
- Change configured-project + live-session merge to be live-session-first.
- Deprecate `project.startup`.
- Keep `project.startup` parseable for compatibility, but ignore it during launch.
- Add Windows-specific default config path support using `%AppData%\zjsh\config.kdl`.
- Make `zjsh config init` emit an OS-appropriate sample `config.kdl`.

## Entry Ordering and `.` Behavior

### Global Display Order

1. live session
2. configured project
3. resurrectable session
4. `.` current-directory entry
5. zoxide path

### `.` Independence

The built-in `.` entry is always independent and never participates in aggregation-time merge.

Rules:

- `.` remains a built-in current-directory entry.
- `.` does not merge with zoxide paths.
- `.` does not merge with configured projects.
- `.` does not merge with any other path-derived or session-derived entry.
- If runtime cwd also appears in zoxide results, both entries remain visible.
- If runtime cwd also matches a configured project path, both entries remain visible.
- If runtime cwd also matches a `cwd true` project's effective path, both entries remain visible.

Purpose: `.` represents the current user context and should remain easy to select even when zoxide returns many path entries.

## Aggregation and Merge

v0.3.2 keeps aggregation-time merge for configured projects and matching Zellij sessions.

### Matching Session

A matching session is any Zellij session whose name equals the configured project's effective session name.

Session name priority:

1. `project.session`
2. `project.name`

### Merge Cases

- configured project + matching live session merge into one entry;
- configured project + matching resurrectable session merge into one entry;
- ordinary zoxide path entries remain independent entries;
- `.` remains independent and never merges.

### Merge Priority

- configured project + live session: live-session-first
- configured project + resurrectable session: project-first

### Live-Session-First Meaning

When a configured project merges with a matching live session:

- the merged entry type is `session`;
- the merged entry name and display identity come from the live session;
- the merged entry still preserves applicable project metadata needed for later connect behavior, such as effective path, layout, layout file, effective shell/default shell behavior, and restart policy;
- parsed `startup` field values may still exist in config-derived metadata, but must not affect launch behavior.

Purpose: if a project already has a running live session, the list should present the running session as the primary thing the user will attach or switch into.

### Project-First Resurrectable Merge

When a configured project merges with a matching resurrectable session:

- the merged entry remains project-first;
- configured project identity remains primary;
- resurrectable session state is preserved on the merged entry.

Purpose: resurrectable state is fallback state, while configured project data remains the primary user intent.

## Launch Behavior

Launch behavior remains as in v0.3.0 except for the `project.startup` rule below.

### `project.startup`

`project.startup` is deprecated in v0.3.2.

Compatibility rule:

- `project.startup` remains parseable in v0.3.2 for config compatibility.
- Existing configs containing `project.startup` must continue to load.

Launch rule:

- `project.startup` is ignored during launch in v0.3.2.
- `project.startup` must not be executed.
- `project.startup` must not be translated into a generated layout.
- `project.startup` must not alter launch behavior.
- A project with only `startup` launches like a plain `path` or `cwd true` project.

Effective launch precedence for configured projects in v0.3.2:

1. `layout_file`
2. `layout`
3. plain project path / `cwd true`

Existing session behavior remains unchanged:

- existing live sessions still attach or switch as before;
- existing resurrectable sessions still use the v0.3.0 `restart_on_resurrection` behavior.

## Config Compatibility

v0.3.2 keeps the v0.3.0 config surface, except that `project.startup` is now deprecated and ignored at launch.

Compatibility rules:

- `project.startup` remains accepted for backward compatibility.
- Unknown fields are still ignored.
- Unknown top-level nodes are still ignored.
- Missing config file still returns default config.

Failure rules remain as in v0.3.0 for correctness-impacting issues such as:

- invalid KDL syntax;
- invalid value type;
- project missing both `path` and `cwd true`;
- project defining both `path` and `cwd true`;
- path expansion failure for static `path` or `layout_file`.

Valid `project.startup` values continue to parse successfully. Invalid value types still follow the normal v0.3.0 parser failure rules.

## Platform Config Path and `config init`

### Default Config Path

Default path by platform:

- Linux and macOS: `~/.config/zjsh/config.kdl`
- Windows: `%AppData%\zjsh\config.kdl`

Rules:

- Windows must not use `~/.config/zjsh/config.kdl` as the default config location.
- Windows should follow the same general config-directory style as Zellij by using the user's roaming application config directory.
- Config loading and `zjsh config init` must use the same default path resolution.
- `zjsh config init` must create the parent directory when needed.

### `config init` Sample

`zjsh config init` must emit an example config that matches the current OS.

Rules:

- Linux and macOS samples may use Unix-style example paths.
- Windows samples should use Windows-style example paths.
- Windows sample paths must be valid KDL strings.
- Windows sample paths must not rely on environment-variable expansion inside config values.
- The emitted sample must remain valid KDL on the current platform.

## Test Requirements

### Ordering

- `.` sorts before ordinary zoxide path entries;
- resurrectable sessions still sort before `.`;
- cwd and matching zoxide path remain separate visible entries;
- cwd and matching configured project path remain separate visible entries;
- ordinary zoxide path entries remain independent entries.

### Aggregation

- configured project + matching live session merges into one live-session-first entry;
- configured project + matching resurrectable session still merges into one project-first entry;
- merged live-session entry preserves project metadata needed for later connect behavior;
- `.` does not merge with matching zoxide paths;
- `.` does not merge with matching configured project paths;
- `.` does not merge with matching `cwd true` project effective paths.

### Launch

- connecting to a project with `layout_file` still uses the layout file;
- connecting to a project with `layout` still uses the named layout;
- connecting to a project with only `startup` ignores `startup` and does not use generated startup-layout execution;
- connecting to a plain path project still uses the project path;
- connecting to a `cwd true` project still uses runtime cwd.

### Config

- existing sample configs still parse successfully;
- configs containing `project.startup` still parse successfully;
- valid `project.startup` values are accepted but ignored at launch;
- invalid `project.startup` value types still fail according to normal parser rules;
- existing v0.3.0 compatibility and failure rules remain unchanged.

### Platform Config

- default config path is `~/.config/zjsh/config.kdl` on Linux and macOS;
- default config path is `%AppData%\zjsh\config.kdl` on Windows;
- `config init` emits a Unix-style sample on Linux and macOS;
- `config init` emits a Windows-style sample on Windows;
- Windows sample config remains valid KDL and does not rely on environment-variable expansion.

## Non-Goals

- Do not hard-reject `project.startup` in v0.3.2.
- Do not remove `restart_on_resurrection` in v0.3.2.
- Do not change v0.3.0 resurrection semantics beyond the live-session-first merge rule.
